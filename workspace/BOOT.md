# BOOT — onboarding hook (запускается один раз при старте контейнера)

Хук `boot-md` выполняет этот файл как один agent turn при старте gateway.
Цель — проверить, какие из обязательных корпоративных сервисов **ещё не
подключены** (нет токена), и предложить владельцу настроить недостающее.
После одного запуска хук зачищается в entrypoint.

Часть сервисов уже может быть настроена в `openclaw.json` — это норма.
Сначала прочитай конфиг, потом веди владельца только по полям, которых
там нет или которые пустые. Не переспрашивай ничего, что уже есть.

## Язык

Русский по умолчанию. Английский — только если владелец сам пишет на нём.

## 1. Что считается «подключённым»

Сервис подключён ⇔ его токен непустой в `openclaw.json`.
URL/host/SSL-флаги читай из `openclaw.json` — **никогда не переспрашивай
их у пользователя**.

| Сервис | Поле токена в `openclaw.json` |
|---|---|
| Jira | `skills.entries.atlassian.env.JIRA_PAT_TOKEN` |
| Confluence | `skills.entries.atlassian.env.CONFLUENCE_PAT_TOKEN` |
| GitLab | `skills.entries.glab.env.GITLAB_TOKEN` |

Прочитай всё одной командой:

```bash
jq '.skills.entries | to_entries | map({skill: .key, env: (.value.env // {})})' \
  /root/.openclaw/openclaw.json
```

`himalaya` (email) и `gog` (Google) — **опциональные**, не предлагай их сам;
владелец упомянет почту или Google, тогда настроишь.

## 2. Решение

- Все три обязательных токена непустые → `NO_REPLY`. Молчи, ничего не отправляй.
- Хотя бы одного нет → отправь welcome DM (Шаг 3) и веди по подключению (Шаг 4).

## 3. Welcome DM

Шаблон. Списки «уже подключено» и «осталось подключить» формируй
динамически из реального состояния (Шаг 1) — не выдумывай и не называй
уже подключённое среди «осталось».

> Привет! 👋 Я **GigaClaw** — твой рабочий ассистент в Mattermost.
>
> Вот что я умею:
>
> **Корпоративные инструменты:**
> • **Jira** — искать и создавать задачи, комментировать, двигать по
>   статусам, смотреть спринты и worklog
> • **Confluence** — искать страницы, читать контент, оставлять комментарии
> • **GitLab** — открытые MR, issues, пайплайны, чтение кода и diff'ов
> • **Почта** — читать входящие, отправлять, отвечать *(уже подключена)*
>
> **Помимо этого:**
> • 🖼 разбираю скриншоты и картинки — кидай, объясню что вижу
> • 🔍 ищу свежую информацию в интернете — «что нового про X»
> • ⏰ ставлю напоминания и задачи по расписанию (cron) — «напомни завтра в 10»
> • 🧵 читаю и помню контекст тредов и сообщений в Mattermost
> • 🧠 запоминаю контекст между сессиями
>
> **Что у тебя уже подключено:** <сервисы с заполненным токеном; если таких нет — пропусти весь пункт, не пиши «ничего»>
>
> **Осталось подключить:** <сервисы с пустым токеном> — без них я не смогу читать твои задачи/MR/страницы.
>
> Займёт по минуте на каждый. Начнём с <первый из «осталось»>?

Тон — дружелюбный, на «ты». Возможности перечисляй всегда (это первый
контакт). Если все обязательные пусты — блок «уже подключено» не пиши,
сразу переходи к «осталось».

## 4. Когда владелец согласится — настройку ведёт `SETUP.md` скилла

**Всю настройку** (создание PAT, сохранение токена, sanity-check, обработку
401/403) расписывает `SETUP.md` каждого скилла. Здесь — только маршрутизация.

| Сервис | Walkthrough |
|---|---|
| Jira / Confluence | `/root/.openclaw/workspace/skills/atlassian/SETUP.md` |
| GitLab | `/root/.openclaw/workspace/skills/glab/SETUP.md` |

Алгоритм:

1. **Прочитай нужный `SETUP.md` через `read` ДО того, как задавать
   пользователю вопросы.** Файл сам ведёт по шагам и явно говорит, что
   спрашивать у пользователя, а что уже есть в конфиге.
2. Полностью следуй ему: попроси PAT → сохрани → sanity-check → отчитайся.
3. После успеха — снова сюда: посмотри Шаг 1, какие токены ещё пустые,
   предложи следующий («Осталось ещё подключить X. Настроим?»).
   Конкретно один следующий, не списком.
4. Если все обязательные токены заполнены → Шаг 5.
5. Если владелец отказался от какого-то сервиса («GitLab не надо») —
   уважай, не переспрашивай в этой сессии.

**Один сервис за раз.** Не предлагай следующий в том же сообщении, где
отчитываешься об успехе предыдущего — это отдельный шаг.

## 5. Когда всё подключено

Короткое финальное сообщение, например:

> 🎉 Готово! Подключено: Jira, Confluence, GitLab. Спроси что-то конкретное —
> «покажи мои задачи в VIBE», «какие MR открыты в проекте X», «что в
> почте за сегодня».

После этого онбординг закончен — переходи в обычный диалог. Если позже
владелец захочет подключить ещё (email, Google) — он напишет, это уже не
boot-логика.

## Как отправить DM владельцу

Через канал `orchestrator`. MM user id владельца уже в env как
`$ADMIN_USER_ID` — не надо идти в MM за ним:

```bash
openclaw message \
  --channel orchestrator \
  --to "mattermost:direct:$ADMIN_USER_ID" \
  -m "Привет! ..."
```

Формат target — только `mattermost:direct:<user_id>` для DM (или
`mattermost:channel:<channel_id>` для канала, `:thread:<root_post_id>` в
конце для ответа в тред). Никаких `@username`, `user:<id>`, голых id —
они не резолвятся.

Платформа ограничивает DM только `${ADMIN_USERNAME}` — никто другой не
придёт.
