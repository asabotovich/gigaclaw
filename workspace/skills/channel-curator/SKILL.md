# Channel Curator

A curator that **keeps a Mattermost channel's open threads moving toward
closure**. Stores a per-project state file, runs morning and evening
check-ins via OpenClaw cron, and proposes write actions (Jira create /
transition / Confluence page) to the channel owner — never performs them
without an explicit `yes`.

> **Group channels only.** Skip if the channel is a DM (`mattermost:direct:...`).
> Curation makes no sense for a 1-to-1 conversation.

---

## Mission — what curation actually delivers

Hold this in your head whenever you make a decision. The skill exists to
give the owner four concrete benefits. Everything else is plumbing.

### 1. Drive open threads to closure  (the central job)

For every open thread the skill asks itself **«what's blocking this from
closing?»** and tries to remove that blocker. Typical blockers and what
to do:

| Blocker | Action |
|---|---|
| No ticket, but should be one | Propose `«завести VIBE-тикет?»` in the thread; create via `jira_create_issue` after `yes` |
| Ticket exists, status stale | `jira_get_issue` for fresh status; if chat consensus disagrees with Jira → propose transition; after `yes` → `jira_transition_issue` |
| Assignee silent >2 days | Ping in thread. After 5 days of silence → escalate to project owner in the next digest |
| Needs a Confluence page | Draft content in the thread, ask for space + `yes`, then `confluence_create_page` |
| Needs an owner decision | Add to `## Decisions pending`, surface in every digest until resolved |
| Blocker unclear | Ask in the thread: «что нужно для закрытия этого треда?» |

Each Open thread carries a **Next step** line that names its blocker. On
every tick you re-read these and act on whatever you can.

### 2. Catch-up without re-reading

Owner returns after 2–3 days, opens `<slug>.md`, and immediately sees:
what closed, what's stuck, who's been pinged, what risks emerged. They
should never need to scroll the channel.

### 3. Don't lose new tasks from chat

When chat contains «@user сделай X к пятнице» — catch it, propose
`«завести VIBE-тикет?»`, create after `yes`. This is the «формирует
задачи в Jira из поручений» requirement.

### 4. Reconcile chat with Jira

Pull fresh status for tracked tickets every morning tick. If Jira ≠
chat-consensus (e.g. Jira `Done`, chat asking «когда зафиксят?») — raise
the discrepancy in the digest.

---

## What the curator does NOT do (MVP)

- **Never write to Jira / Confluence without explicit `yes` in the
  thread.** Not "implied consent" from `:+1:`, not "the owner usually
  agrees". Always a fresh `yes` for each write.
- Never hallucinate context. If the sweep only got part of the channel
  history, say so («у меня X% истории, не уверен») rather than making
  things up.
- Never spam. **One ping per thread per day, max.** If digest blocks
  would all be empty, don't post the digest.
- Never touch another channel — only the channel attached to this
  project.
- No proactive advisory («как сократить сроки», «как снизить риск с N
  до M»). Post-MVP.

## Calling sequence — three scenarios

When activated, classify the current invocation into one of three
scenarios. They never overlap.

| Scenario | Trigger | Distinguishing condition |
|---|---|---|
| **A. Onboarding** | Owner says «веди этот канал», «возьми проект», «суммаризируй и следи» | Channel has **no** entry in `index.md`, or `curator_active: false` |
| **B. Tick** | Cron-fired prompt («Утренний/Вечерний чек проекта X») **or** owner says «проверь», «актуализируй» | Project file exists, `curator_active: true` |
| **C. Offboarding** | «хватит», «останови», «отключи кураторство» | Project file exists, prompt contains stop intent |

To classify: read `index.md`, find row by `channel_id`, check
`curator_active`, then look at prompt wording. If ambiguous — ask the
owner before doing anything.

## Scenario A — Onboarding

### A.1 Identify the project

1. `channel_id` comes from session context (strip leading `#`).
2. `project_slug`: take from explicit owner instruction («веди этот
   канал как `openclaw-pilot`») or derive kebab-case from the channel
   label.
3. If `index.md` already has the slug and `curator_active: true` →
   pivot to Scenario B silently.

### A.2 Size the channel (always, before sweeping)

This is the safety gate against megachannels. Hit MM through the
orchestrator's `/read/*` proxy (same as `mattermost` skill):

```bash
# Channel stats: total_msg_count, member_count
curl -sf -X POST \
  -H "Authorization: Bearer $ORCHESTRATOR_PUSH_SECRET" \
  -H "Content-Type: application/json" \
  -d "{\"channel_id\":\"$CHANNEL_ID\"}" \
  "$ORCHESTRATOR_URL/read/channel-stats"

# Oldest post timestamp: read first page sorted ASC, or paginate
# backward until empty. The first page from /read/channel-history
# returns newest-first; to find the oldest, paginate with `before`
# until the response is empty.
```

(If `/read/channel-stats` isn't available, fall back to counting via
pagination — but `total_msg_count` is what we want.)

### A.3 Pick a sweep strategy

| Channel size | Behaviour |
|---|---|
| Small (≤ ~2000 posts, ≤ ~90 days) | Full sweep, no question |
| Medium (~2k–20k posts or 90–365 days) | Post a summary to the channel: «канал содержит N постов за M месяцев, K участников. Какой scope взять — (a) last_30d, (b) active_threads_only, (c) full?». Wait for the owner |
| Large (>20k or >1 year) | **Do not sweep.** Ask the owner to name a scope (period / keywords / their threads only). Without explicit scope — decline |
| Owner silent ≥10 min on a medium channel | Default to `last_30d` and announce «начал с последних 30 дней, скажи если надо больше» |

Save the chosen value into `sweep_strategy` and `channel_size_at_init`
in the project file.

### A.4 Sweep, in batches

For `full` and `last_30d` strategies:

- Paginate `POST /read/channel-history {channel_id, per_page: 200, before: <oldest-seen-post-id>}` backward.
- **Batch limit: ~10 pages (≈2000 posts) per tick.** Save the
  intermediate `last_seen_post_id` to the project file as you go. If
  you hit the limit, post «продолжу следующим тиком» to the channel
  and let the next cron tick continue.
- For each root post with `reply_count > 0`, fetch the full thread
  (`/read/thread` is unpaginated).

For `active_threads_only`:
- Fetch the most-recent page, find all root posts in the last N days,
  fetch each thread.

For `custom`: follow the scope the owner gave.

### A.5 Analyse (use a sub-agent with extended thinking)

Hand the transcript to a sub-agent. Ask it to produce:

- 2–4 lines: what this channel is about, what's going on, who's the
  owner
- A list of **open threads** as free-form cards. Each card MUST have a
  `Next step` line — name the concrete blocker
- **Tasks**: VIBE-XXX (or similar) mentioned anywhere in chat. Enrich
  each via `jira_search` / `jira_get_issue` (`atlassian` skill, see
  below)
- **Decisions pending**: open questions to the owner
- **Risks**: explicit fears / blockers voiced in chat

> Do **not** invent participants list — it's available live via MM API,
> we don't duplicate it.

### A.6 Write project file + index

Create `~/.openclaw/workspace/skills/channel-curator/projects/<slug>.md`
with the frontmatter (filled-in) and all sections (Open threads, Tasks,
Decisions pending, Risks, Closed threads).

Append a row to `index.md`:

```
- [<slug>](projects/<slug>.md) — <channel label> — owner @<username>
```

### A.7 Greet the channel

One root post to the channel (not inside any thread):

```
Взял <slug> под кураторство.
Прочитал N постов за период T (стратегия: <strategy>).
Вижу: K активных тредов, M задач, L открытых вопросов, R рисков.
<если есть>Уточни:
- <question 1>
- <question 2>
```

### A.8 Register cron jobs

```bash
SLUG="openclaw-pilot"
CHANNEL_ID="5ic9mo3337d93mwachkdmcd8oo"
TARGET="mattermost:channel:$CHANNEL_ID"

MORNING=$(openclaw cron add \
  --name "curator-${SLUG}-morning" \
  --cron "0 9 * * *" --tz "Europe/Moscow" \
  --session isolated \
  --message "Утренний чек проекта ${SLUG}. Прочитай ~/.openclaw/workspace/skills/channel-curator/projects/${SLUG}.md и действуй по инструкции skill'а channel-curator (Scenario B — Tick)." \
  --announce --channel orchestrator --to "$TARGET")

EVENING=$(openclaw cron add \
  --name "curator-${SLUG}-evening" \
  --cron "0 18 * * *" --tz "Europe/Moscow" \
  --session isolated \
  --message "Вечерний чек проекта ${SLUG}. Прочитай ~/.openclaw/workspace/skills/channel-curator/projects/${SLUG}.md и действуй по инструкции skill'а channel-curator (Scenario B — Tick)." \
  --announce --channel orchestrator --to "$TARGET")

MORNING_ID=$(printf '%s' "$MORNING" | python3 -c "import json,sys; print(json.load(sys.stdin)['jobId'])")
EVENING_ID=$(printf '%s' "$EVENING" | python3 -c "import json,sys; print(json.load(sys.stdin)['jobId'])")
```

Write `MORNING_ID` and `EVENING_ID` into the frontmatter
(`morning_cron_id`, `evening_cron_id`).

## Scenario B — Tick (morning / evening / on-demand)

A tick is four phases. Each phase serves one of the four Mission
bullets.

### Phase 1 — Ingest

```bash
# Pull new posts strictly after last_seen_post_id, paginated, capped.
# Use /read/channel-history with `after=<last_seen_post_id>`.
# Limit: ≤ 500 new posts per tick.
# If response is bigger, process oldest 500, leave last_seen pointing
# mid-batch, mention the remainder in the digest.
```

### Phase 2 — Update state (Mission #2, #4)

For each new post:

- **In an existing open thread of this project** → refresh that thread's
  card: update Last activity, rewrite summary if the conversation
  shifted meaningfully, move `Next step` if the answer arrived. If the
  thread looks resolved (final agreement / fix landed / decision
  recorded) — move card to `## Closed threads` with one line «closed
  YYYY-MM-DD — итог: …».
- **Root post of a new thread** → add a card to `## Open threads` with
  a tentative summary and `Next step: ?`. Surface it in the digest as
  «новый тред — включить в проект?».
- **Mentions an existing `<PROJECT>-NNN` ticket** → `jira_get_issue`,
  refresh the `## Tasks` row. **Reconciliation rule (Mission #4):** if
  Jira status disagrees with the chat consensus (e.g. Jira `Done`,
  chat «когда зафиксят?») → mention in the digest.
- **Looks like a new explicit task** (e.g. «@user сделай X к пятнице»,
  «надо чтобы кто-то сделал Y») → handled in Phase 4.
- **Voices a risk** (markers: «боюсь», «опасаюсь», «есть риск», «не
  уверен что») → add a line to `## Risks` with the quote. If a
  mitigation was proposed in the same thread, attach it.
- **Records a decision** («решили делать X», «договорились что Y») →
  close the matching `## Decisions pending` item if any.

Additionally, **once per day** (morning tick only): walk all rows of
`## Tasks` and `jira_get_issue` each one, even if it wasn't mentioned
in chat. Refresh the table. Carry status changes into the digest.

### Phase 3 — Drive threads to closure (Mission #1)

This is the heart of the skill. Iterate `## Open threads`. For each,
look at its `Next step` and act:

| Next step shape | Action |
|---|---|
| «дождаться X от @user», `(now − last_activity) > 2d`, no curator ping today | Post in thread «@user, есть апдейт?». Update `Last pinged by curator` |
| Same as above, no answer ≥5 days since the first curator ping | Surface in the digest's `🆙 Эскалация` block — owner needs to step in |
| «нужно решение про X» | Make sure it's also in `## Decisions pending`. If >7 days old without movement — mark in digest |
| «завести тикет VIBE» | Post in thread «нашёл задачу: …. Завести VIBE с owner @<user>, deadline <date>? yes/no» → on `yes` call `jira_create_issue` (see code below) |
| «закрыть VIBE-XXX» / «перевести в Done» | Post «по чату это уже Done. Перевести VIBE-XXX в Done? yes/no» → on `yes` call `jira_transition_issue` |
| «нужна страница в Confluence», «задокументировать X» | Draft the page body and post «вот draft страницы про X (текст). Создать в space <Y>? yes/no» → on `yes` call `confluence_create_page` |
| «обновить Confluence-страницу Z» | Post the proposed diff and ask `yes/no`; then `confluence_update_page` |
| Blank or unclear | Post in thread: «что сейчас нужно для закрытия этого треда?» — once, never repeat for the same thread |

**Hard rules:**

- One curator post per thread per day, max. Track via `Last pinged by curator`.
- Every write to Jira / Confluence requires a fresh `yes`. No implicit consent.
- If multiple proposals on the same tick — bundle into one message in
  the same thread so the owner can answer in one go.

### Phase 4 — Digest post (Mission #2 + #3)

**Task detection from new posts.** A regex / heuristic pass over new
posts to spot patterns like:
- `@<user> ... (сделай|реализуй|возьми|сделать) ... (к|до|до пятницы|завтра|...)`
- «надо чтобы кто-то X»
- «давайте сделаем Y»

For each candidate, post in the thread (or in-channel if not threaded):
«нашёл поручение: <text>. Завести как тикет VIBE? Owner: @<user>,
deadline: <date>. Ответь yes/no — можешь поправить summary.»

On `yes` in the same thread → `jira_create_issue`, attach link in
thread + in `## Tasks`.

**Digest post**, one root post to the channel:

```
Чек проекта <slug> — <date> <утро|вечер>

За <период с last_swept_at>: N постов, K тредов

✅ Закрыто: <thread / task>, …
🟡 В работе: <task / thread> — @owner — статус, Next step: …
⚠️ Висит >2 дней: <thread / task> — ждём @user, пингнул сегодня
🆙 Эскалация owner'у: <thread> — @user молчит >5 дней
❓ На подтверждение: <M новых поручений / страниц / транзитов — ссылки на треды>
⚡ Расхождения чата и Jira: <key: Jira=<status>, чат обсуждает как <other>>
🤔 Decisions pending >7 дней: <list>
🆕 Новые риски: <short list>
```

**Skip rules:**
- Any empty block → omit it.
- **All blocks empty → don't post the digest.** A silent tick is the
  correct outcome when nothing happened.

### Phase 5 — Persist

- Update `last_seen_post_id` and `last_swept_at` in the frontmatter.
- If Phase 1 hit the 500-post cap, also note «не разобрал X постов,
  доберу следующим тиком» in the digest (which is the only time you
  post even with empty blocks).

## Scenario C — Offboarding

When the owner says «хватит / останови / отключи кураторство»:

```bash
SLUG="openclaw-pilot"
PROJECT_FILE="$HOME/.openclaw/workspace/skills/channel-curator/projects/${SLUG}.md"
INDEX="$HOME/.openclaw/workspace/skills/channel-curator/index.md"

# 1. Read cron job ids from frontmatter
MORNING_ID=$(grep -E '^morning_cron_id:' "$PROJECT_FILE" | awk '{print $2}')
EVENING_ID=$(grep -E '^evening_cron_id:' "$PROJECT_FILE" | awk '{print $2}')

# 2. Remove cron jobs
[ -n "$MORNING_ID" ] && openclaw cron rm "$MORNING_ID" 2>/dev/null || true
[ -n "$EVENING_ID" ] && openclaw cron rm "$EVENING_ID" 2>/dev/null || true

# 3. Mark inactive in frontmatter
sed -i 's/^curator_active: true$/curator_active: false/' "$PROJECT_FILE"

# 4. Mark archived in index (or remove the row)
sed -i "s|^- \[${SLUG}\](.*|- [${SLUG}](projects/${SLUG}.md) — archived|" "$INDEX"
```

Keep the project file — owner may reactivate, and the history is
useful.

## Failure modes to watch

- **Stuck mid-sweep.** If a tick crashed mid-sweep, on the next tick
  `last_seen_post_id` is set, so the resume path is the same as
  incremental. Just keep going from where you left off.
- **Owner answers with `yes` for an action that no longer applies.**
  E.g. owner said `yes` to creating a ticket, but in the same minute
  someone else closed the thread by other means. Look at the current
  state of the thread before acting on stale confirmations. If unsure
  — ask again.
- **Channel deleted / bot removed.** `/read/channel-stats` will fail.
  In Scenario B, if three consecutive ticks fail to read the channel
  — auto-deactivate (Scenario C) and DM the owner about it.

## Activation checklist

When this skill starts, before doing anything else:

1. Source the env vars:
   `ORCH="$ORCHESTRATOR_URL"; ORCH_AUTH="Bearer $ORCHESTRATOR_PUSH_SECRET"`
2. Confirm channel kind is `group_channel` (or thread inside one) — if
   `direct`, decline and explain why.
3. Run the path-helper block from "Where state lives".
4. Classify scenario (A / B / C).
5. Proceed by the relevant section above.
