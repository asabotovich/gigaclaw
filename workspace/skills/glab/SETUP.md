# GitLab — настройка (walkthrough)

Читай этот файл **только** когда владелец впервые подключает GitLab (или
меняет токен). Для обычной работы смотри `SKILL.md`.

## Перед стартом

Сначала прочитай что уже есть в конфиге:

```bash
jq '.skills.entries.glab.env' /root/.openclaw/openclaw.json
```

Ожидаемые поля:

- `GITLAB_HOST` — заполнено заранее из `.env` хоста (например, `git.sberdevices.ru`)
- `GITLAB_TOKEN` — заполняет пользователь (его мы и подключаем)

Если `GITLAB_HOST` отсутствует — Оркестратор его не прокинул.
Останавливайся, говори пользователю написать админу; хост не гадай.

## Шаг 1 — Попроси пользователя создать PAT

Сначала прочитай хост (нужен для ссылки):

```bash
GITLAB_HOST=$(jq -r '.skills.entries.glab.env.GITLAB_HOST // empty' /root/.openclaw/openclaw.json)
```

Отправь пользователю DM:

> Подключаем GitLab. Делаем так:
>
> 1. Открой **https://$GITLAB_HOST/-/user_settings/personal_access_tokens**
> 2. Жми **Add new token**:
>    • Name: `gigaclaw`
>    • Expiration date: ставь максимум (обычно +1 год)
>    • Scopes: обязательно **`read_api`** и **`read_repository`**
>      (этого хватит для поиска MR/issues и чтения файлов; если захочешь
>      позже, чтобы я умел создавать и комментировать MR — добавь ещё
>      `api` и `write_repository`, это опционально)
> 3. Жми **Create personal access token**. Откроется страница с токеном.
> 4. **Скопируй токен сразу** — GitLab показывает его один раз, потом только
>    его хеш. Пришли мне следующим сообщением.

### Scopes — краткая шпаргалка

| Scope | Зачем |
|---|---|
| `read_api` | Искать MR, issues, проекты — **обязательно** |
| `read_repository` | Читать файлы, коммиты, ветки — **обязательно** |
| `api` | Создавать MR, комментировать, approve — опционально |
| `write_repository` | Push-операции — опционально, используй только если точно нужно |

Для пилота достаточно read-only, бот всё равно ничего лишнего не сделает.

## Шаг 2 — Сохрани токен

```bash
openclaw config set skills.entries.glab.env.GITLAB_TOKEN "<значение-от-пользователя>"
```

**Не повторяй токен в ответе.**

## Шаг 3 — Sanity-check (обязательно!)

```bash
CFG=/root/.openclaw/openclaw.json
export GITLAB_HOST=$(jq -r '.skills.entries.glab.env.GITLAB_HOST // empty' "$CFG")
export GITLAB_TOKEN=$(jq -r '.skills.entries.glab.env.GITLAB_TOKEN // empty' "$CFG")

# Кто я? (заодно валидирует авторизацию)
glab api user
```

Если вернулся JSON-объект с `username` / `name` — авторизация работает.

Отчитайся пользователю с **реальным именем и username** из ответа:

> ✅ GitLab подключён. Ты — `<username>` (<real name>).
>
> Готов искать MR, issues, читать файлы. Скажи что-то вроде:
> • «покажи открытые MR в проекте sberdevices-frontend/vibe-projects»
> • «что нового в issues у меня назначенных»
> • «посчитай мои коммиты за неделю»

### Если Шаг 3 упал

`glab api user` с PAT — штатная проверка авторизации. 401/403 →
токен не тот, другие схемы (OAuth, пароли) не пробуй. Просто попроси
пересоздать:

- **401 Unauthorized** → скорее всего токен не полностью скопирован или
  истёк. Скажи пользователю: «Токен не принят GitLab — попробуй создать
  новый, скопируй целиком».
- **403 Forbidden** (редко для своего же `/user`) → токен без минимальных
  scope'ов. Попроси создать заново с `read_api`.
- **Connection refused / DNS fail** → бот не видит GitLab из сети.
  Для корпоративного GitLab — скорее всего на хосте бота нужна VPN.
  Это не проблема токена, это инфра.

## После успешного подключения

Когда GitLab подключён, **не заканчивай онбординг сам**. Вернись к
`/root/.openclaw/workspace/BOOT.md` Шаг 4 и сделай так:

1. Прочитай `openclaw.json` ещё раз и посмотри, какие **обязательные**
   токены (`JIRA_PAT_TOKEN`, `CONFLUENCE_PAT_TOKEN`, `GITLAB_TOKEN`) ещё
   пустые.
2. Если остались — предложи пользователю один конкретный следующий
   сервис (не списком, а конкретикой). Например:
   > Ещё осталось подключить Jira. Настроим сейчас?
3. Если согласится — иди в нужный `SETUP.md` этого скилла.
4. Если откажется («потом настрою») — запомни отказ в рамках сессии,
   не переспрашивай. Переходи в обычный диалог.
5. Если все обязательные токены заполнены — отправь финальное сообщение
   из `BOOT.md` Шаг 5, онбординг закончен.

**Не предлагай следующий сервис в том же сообщении, где отчитываешься об
успехе** — это отдельный шаг после подтверждения пользователем.

## Важно

- После сохранения скилл работает уже на следующем вызове через
  jq+export-паттерн из `SKILL.md` — **перезапуск шлюза не нужен**.
- Ротация токена: когда текущий истекает, просто повтори Шаги 1–3.
  `openclaw config set` перезапишет старое значение.
